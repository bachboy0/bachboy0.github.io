# ADR-0002: Meta-delivered CSP strategy for static hosting constraints

**Date:** 2026-03-02  
**Status:** Accepted

## Context

This project is deployed as a static site to GitHub Pages using `.github/workflows/deploy.yml`. In this repository setup, security controls are applied at document level in `src/components/BaseHead.astro`.

The site requires a small inline script for theme initialization (FOUC prevention), and local Astro/Vite development tooling injects resources that differ from production behavior.

A practical security strategy is needed that:

- keeps production policy strict,
- keeps local development usable,
- and remains compatible with static hosting constraints.

## Decision

We will enforce Content Security Policy through a `<meta http-equiv="Content-Security-Policy">` tag generated in `src/components/BaseHead.astro`, with environment-specific policy values.

- **Development policy** (`import.meta.env.DEV`): allows inline styles and eval needed by local tooling.
- **Production policy**: keeps stricter style policy (`style-src 'self'`) and allows only required inline scripts.

We will also keep referrer hardening using:

- `<meta name="referrer" content="strict-origin-when-cross-origin">`

We will not claim unsupported protections as active in this implementation. Specifically, invalid or non-effective meta-based header claims are removed from active docs and implementation statements.

## Consequences

### Positive

- Production pages keep stronger CSP behavior.
- Local development avoids excessive CSP breakage from dev tooling internals.
- Security documentation now matches actual implementation.

### Negative / Trade-offs

- Meta-delivered CSP is weaker than true HTTP response headers.
- Inline script allowance remains a trade-off for theme initialization behavior.
- Security posture differs between dev and production by design, which requires contributor awareness.

## Alternatives Considered

| Option | Why not chosen |
| --- | --- |
| Single strict CSP for all environments | Breaks local development tooling with repeated inline-style/script violations |
| Broadly allow `'unsafe-inline'` in production styles/scripts | Reduces production CSP effectiveness more than necessary |
| Full nonce/hash-based CSP architecture | Higher complexity than justified for this static site’s current scope |

## Evidence

- `src/components/BaseHead.astro` (environment-specific `cspContent` using `import.meta.env.DEV`)
- `src/pages/index.astro` (inline script usage patterns)
- `.github/workflows/deploy.yml` (static deploy to GitHub Pages)
- `SECURITY.md` (current policy statement)
