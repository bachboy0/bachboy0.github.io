# ADR-0002: Security Headers via `<meta http-equiv>` in BaseHead

**Date:** 2026-03-02 **Status:** Accepted

---

## Context

The site is a fully static build hosted on **GitHub Pages**. GitHub Pages serves files directly from a CDN and provides **no mechanism to set custom HTTP response headers**. Standard browser security features — Content Security Policy, `X-Frame-Options`, `X-Content-Type-Options`, etc. — are normally delivered as HTTP response headers, which is the most reliable and spec-compliant approach.

Without these controls the site would be exposed to:

- **Clickjacking** — the page could be embedded in a malicious `<iframe>`
- **MIME sniffing attacks** — browsers could misinterpret content types
- **Referrer leakage** — full URLs could be sent to third-party origins
- **Unnecessary browser API access** — geolocation, camera, microphone accessible by default

Additionally, the site includes a small inline `<script>` for FOUC (Flash Of Unstyled Content) prevention during theme initialisation, which must be accounted for in any CSP configuration.

## Decision

We will deliver all achievable security controls as `<meta http-equiv>` tags inside `src/components/BaseHead.astro`, which is included on every page.

| Control | Implementation | Value |
| --- | --- | --- |
| Content Security Policy | `<meta http-equiv="Content-Security-Policy">` | `default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self'; img-src 'self' data: https:; font-src 'self'; connect-src 'self'; frame-ancestors 'none';` |
| Clickjacking | `<meta http-equiv="X-Frame-Options">` | `DENY` |
| MIME sniffing | `<meta http-equiv="X-Content-Type-Options">` | `nosniff` |
| Referrer leakage | `<meta name="referrer">` | `strict-origin-when-cross-origin` |
| Browser API access | `<meta http-equiv="Permissions-Policy">` | `geolocation=(), microphone=(), camera=()` |

**CSP rationale by directive:**

- `script-src 'unsafe-inline'` — required for the theme-init inline script (`is:inline`) that prevents FOUC; there is no nonce mechanism available in Astro's static output without a custom Vite plugin.
- `style-src 'self'` — Tailwind CSS 4 generates all styles at build time into an external stylesheet; no inline styles are injected at runtime, so `'unsafe-inline'` is intentionally omitted.
- `frame-ancestors 'none'` — CSP equivalent of `X-Frame-Options: DENY`; both are included for broad browser compatibility.
- `img-src https:` — permits Open Graph preview images fetched from arbitrary HTTPS origins by external services.

Beyond the headers, the site also reduces its attack surface by design:

- No external JavaScript (no analytics, no CDN-hosted libraries)
- Self-hosted fonts (no requests to `fonts.googleapis.com` or similar)
- No forms, APIs, or user data collection

## Consequences

### Positive

- All major clickjacking, MIME sniffing, and referrer-leakage vectors are mitigated within the constraints of the hosting platform.
- Zero runtime cost — headers are static HTML baked into the build.
- No additional dependencies or build steps required.
- `style-src` excludes `'unsafe-inline'`, which is a stronger posture than many Tailwind sites that require it.

### Negative / Trade-offs

- `<meta http-equiv>` tags are **processed by the browser's HTML parser**, not by the HTTP layer. They are ignored for navigations that never reach the parser (e.g. prefetch responses, certain service worker intercepts). True HTTP headers would be more reliable.
- `script-src 'unsafe-inline'` weakens the XSS mitigation of CSP. A nonce-based approach would be stricter but requires build-time instrumentation not currently justified.
- `X-Content-Type-Options` and `Permissions-Policy` via `<meta>` have **limited or no browser support** compared to their HTTP header equivalents; they are included for forward-compatibility and documentation purposes.
- If the hosting platform changes in the future (see Alternatives), these meta tags should be replaced with proper HTTP headers.

---

## Alternatives Considered

| Option | Reason not chosen |
| --- | --- |
| Migrate to Netlify / Vercel for custom headers | Adds deployment complexity and breaks the existing GitHub Actions + GitHub Pages pipeline; not justified for a personal blog |
| Cloudflare proxy in front of GitHub Pages | Adds a third-party dependency and ongoing configuration overhead; overkill at this scale |
| Nonce-based CSP to eliminate `unsafe-inline` | Requires a Vite/Astro plugin to inject per-request nonces into the inline theme script; significant complexity for one small script |
| Omit security headers entirely | Leaves the site unnecessarily exposed; meta-tag headers are imperfect but meaningfully better than nothing |
