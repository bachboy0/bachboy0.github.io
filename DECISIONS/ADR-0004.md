# ADR-0004: Adopting a GitHub Copilot-Oriented AI Coding Agent Development Workflow

**Date:** 2026-03-02 **Status:** Accepted

---

## Context

AI Coding Agents such as GitHub Copilot (Chat / Agent mode) significantly accelerate code generation, refactoring, and documentation, but introduce the following challenges:

- The AI cannot grasp project-specific context (tech stack, conventions, past decisions) and produces inaccurate suggestions.
- Prompts are not reused, so the same instructions must be rewritten from scratch every session.
- Architecture and design decisions become tacit knowledge, consulted by neither the AI nor human reviewers.

A structured workflow was needed to address these issues and enable continuous improvement in human-AI collaboration.

## Decision

We will establish a GitHub Copilot-oriented development workflow built on the following four pillars:

### 1. Maintaining thorough project documentation (`ARCHITECTURE.md` / `SECURITY.md` / `README.md`)

`ARCHITECTURE.md` documents the component structure, routing, i18n design, and build & deploy flow. `SECURITY.md` defines the security policy and vulnerability reporting procedure. `README.md` serves as the primary reference for project overview, setup instructions, and the development workflow.

Keeping these documents up to date ensures the AI can obtain structural context before reading any code.

### 2. Placing and operating `.github/copilot-instructions.md`

`.github/copilot-instructions.md` describes the tech stack, coding conventions, common patterns, and how to reference Astro documentation. GitHub Copilot automatically reads this file as a system prompt, providing project-specific context to the AI without repeating it in every session.

Contents include:

- Tech stack (Astro 5, Tailwind CSS 4, MDX, TypeScript)
- Project structure and the role of each directory
- Implementation patterns for components, Content Collections, and styling
- Instruction to prioritise Astro documentation and use the `mcp_astro_docs_search_astro_docs` tool
- Overview of the deployment flow

### 3. Accumulating and leveraging ADRs in the `DECISIONS/` directory

Architecture Decision Records (ADRs) are stored in the `DECISIONS/` directory using a sequential naming scheme (`ADR-XXXX.md`). `DECISIONS/ADR.md` acts as an index so all decisions can be reviewed at a glance.

By referencing ADRs in AI prompts (e.g. "taking past decisions into account"), the AI can factor in recorded constraints and rationale when making suggestions.

### 4. Placing reusable prompts in `.github/.prompts/`

Frequently used instructions, checklists, and task templates are stored as Markdown files in `.github/.prompts/` and invoked via the `#` file-reference feature in GitHub Copilot Chat.

Currently available prompts:

| File | Purpose |
| --- | --- |
| `add-documents.md` | Guidance and steps for updating documentation when adding a new feature |
| `security-checklist.md` | Security-focused review checklist |

## Consequences

### Positive

- `copilot-instructions.md` removes the need for the AI to re-learn project structure every session, yielding more consistent suggestions.
- ADRs live in the same repository, so both AI and humans consult the same source to understand "why was this implemented this way".
- Reusable prompts decouple the quality of routine tasks (documentation updates, security reviews) from the skill of the individual writing the prompt.
- Maintaining project documentation promotes explicit articulation of design intent, which also raises the quality of human code reviews.

### Negative / Trade-offs

- If `copilot-instructions.md` or ADR content becomes stale, the AI may operate under incorrect assumptions. Discipline is required to keep documents current.
- Final judgement on AI suggestions must remain with humans; review processes cannot be skipped to guard against AI-induced oversights.
- The growing number of prompt files may require a formal deprecation workflow for prompts that are no longer needed.

---

## Alternatives Considered

| Option | Reason not chosen |
| --- | --- |
| Typing instructions into the chat ad hoc | Not reproducible; prompt quality varies by session and contributor |
| Relying on README alone without `copilot-instructions.md` | README is written for human readers and serves a different purpose from structured context intended for AI consumption |
| Managing ADRs in an external wiki (e.g. Confluence) | Separating records from the codebase prevents the AI from obtaining context from the repository alone |
| Saving prompts in local notes | Prevents sharing with collaborators and direct reference by AI tooling |
