# ADR-0001: Manual File-Based i18n Without an Astro Integration

**Date:** 2026-03-02 **Status:** Accepted

---

## Context

This site targets three locales — English (`en`), Japanese (`ja`), and Korean (`ko`) — with English as the default.

Astro does not ship a built-in i18n integration that covers every case; the `i18n` config added in Astro 3 helps with routing but leaves UI-string translation completely to the developer. Third-party libraries (e.g. `astro-i18next`, `paraglide-astro`) add abstraction in exchange for additional runtime dependencies, build complexity, and lock-in to an external project's release cadence.

Because the site is a static personal blog with a small, stable set of UI strings, the translation surface is limited and changes rarely.

## Decision

We will implement i18n manually using two thin utility modules and Astro's file-based routing, with no external i18n library.

| Concern | Implementation |
| --- | --- |
| Supported locales & labels | `src/i18n/ui.ts` — `languages` map and `defaultLang` constant |
| UI string catalogue | `src/i18n/ui.ts` — `ui` object keyed by locale → translation key |
| Runtime helpers | `src/i18n/utils.ts` — `useTranslations`, `getLangFromUrl`, `getLocalizedPath`, `removeLocalePath` |
| Routing | Astro file-based routing: default locale at `/`, non-default at `/{locale}/` |
| Blog content | `lang` frontmatter field on each post; `getStaticPaths` filters by locale per route tree |

The default locale (`en`) is served without a URL prefix; non-default locales prefix with their code (e.g. `/ja/blog/`).

Type safety is achieved by exporting `Locale` and `TranslationKey` from `ui.ts`, so callers that pass unknown keys or locales fail at build time.

## Consequences

### Positive

- Zero additional runtime dependencies; no third-party library to maintain or update.
- Trivially understandable — the entire i18n layer is ~80 lines across two files.
- Adding a new locale requires only: a new key in `languages`, a new entry in `ui`, and a new directory under `src/pages/`.
- Type-safe: invalid locale codes and missing translation keys are caught at compile time.
- Build output is fully static; locale routing adds no server overhead.

### Negative / Trade-offs

- No automatic fallback UI for missing translations beyond the one-liner in `useTranslations` (falls back to `defaultLang`).
- All UI strings must be added manually to every locale in `ui.ts`; there is no extraction tooling or translation file format (e.g. YAML, JSON, PO) that external translators could use.
- Pluralisation, number formatting, and date localisation are not handled by this layer and must be managed ad hoc.

---

## Alternatives Considered

| Option | Reason not chosen |
| --- | --- |
| Astro `i18n` routing config (`astro.config.mjs`) | Handles URL rewriting only; does not solve UI-string translation, so `ui.ts` would still be needed anyway |
| `astro-i18next` | Adds `react-i18next` semantics and a runtime dependency; overkill for a static site with a small string catalogue |
| `inlang` / Paraglide | Excellent tooling but heavy setup (SDK, compiler plugin, message format); not justified at this project scale |
| Separate `.json` locale files | More portable for external translators, but adds a loader step with no practical benefit while the author manages all three locales |
