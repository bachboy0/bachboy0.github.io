# ADR-0003: Containerizing the Development Environment with Docker

**Date:** 2026-03-02 **Status:** Accepted

---

## Context

The project previously assumed a locally installed Node.js / npm on the developer's machine, which introduced environment-specific variations (Node version, OS, path configuration, etc.) that could lead to inconsistent build results and increased onboarding costs.

Additionally, first-class tooling support exists for Docker-based development environments in VS Code Dev Containers and GitHub Codespaces, enabling any developer to start working without polluting their host machine.

## Decision

We will containerize the development and build environment using Docker, managed through the following four configuration files:

| File | Role |
| --- | --- |
| `Dockerfile` | Defines the container image based on `node:lts`, installs dependencies, and runs the build |
| `compose.yaml` | Declares the `astro` dev service with source bind-mount, port forwarding, and SSH Agent socket forwarding |
| `.devcontainer/devcontainer.json` | VS Code Dev Containers / Codespaces configuration. Reuses the `astro` service from `compose.yaml` and declares recommended extensions, editor settings, and port forwarding |
| `.dockerignore` | Excludes `node_modules/`, `dist/`, `.env*`, and other unnecessary paths from the build context |

**Key design decisions:**

- Source code is bind-mounted to `/app` via `compose.yaml` so hot-reload works without rebuilding the container image.
- `node_modules` is isolated in a named Docker volume to avoid conflicts with the host OS.
- The SSH Agent socket (`${HOME}/.ssh/ssh-agent.sock`) is mounted into the container, allowing transparent SSH authentication from inside.
- A custom bridge network (`bachboy0-network`) with MTU set to `1200` is used to work around packet fragmentation issues in some VPN environments.
- `devcontainer.json` declares extensions (Astro, Tailwind CSS, Copilot, etc.) and editor settings so that a consistent development experience is available immediately after checkout.

## Consequences

### Positive

- No local Node.js installation is required; setup is reduced to a single `docker compose up`.
- Build reproducibility issues caused by Node version or OS differences are eliminated.
- Dev Containers automatically apply extensions and editor settings, turning the repo into a ready-to-code environment from the moment it is cloned.
- The same container definition works with GitHub Codespaces.

### Negative / Trade-offs

- Docker Desktop (or an equivalent runtime) must be installed on the developer's machine, adding initial setup overhead for those who do not have it.
- A bind-mount configuration can exhibit degraded file I/O performance on Docker Desktop for macOS and Windows.
- The SSH Agent socket path (`${HOME}/.ssh/ssh-agent.sock`) is host-dependent and may need adjustment in environments with different socket configurations.

---

## Alternatives Considered

| Option | Reason not chosen |
| --- | --- |
| Direct host installation (no Docker) | Cannot eliminate environment differences or guarantee build reproducibility |
| Pure devcontainer setup using Dev Container Features (`ghcr.io/devcontainers/features/node`) | Loses fine-grained network and volume control provided by `compose.yaml` |
| GitHub Codespaces-only configuration | Forfeits the benefit of using the same container for local development |
