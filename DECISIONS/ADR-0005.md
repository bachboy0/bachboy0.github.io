# ADR-0005: Explicit `/en/` URL Prefix for All Locales and Standalone Root Landing Page

**Date:** 2026-03-04 **Status:** Accepted

> Supersedes the routing clause of [ADR-0001](./ADR-0001.md) which stated that the default locale (`en`) is served without a URL prefix.

---

## Context

The original i18n routing design (ADR-0001) placed the English default locale at the root (`/blog/`, `/about`) while non-default locales (`ja`, `ko`) used explicit prefixes (`/ja/blog/`, `/ko/blog/`). This asymmetry caused:

- Inconsistent file organisation: `src/pages/blog/` and `src/pages/about.astro` existed at the root level, while `src/pages/ja/` and `src/pages/ko/` grouped locale content under a directory.
- Asymmetric `getStaticPaths()` slug logic: English routes used raw `post.id` while Japanese/Korean routes stripped a locale suffix, making the three implementations diverge unnecessarily.
- `getLocalizedPath()` requiring a special case for `defaultLang` that was only relevant for navigating to English routes.
- The root `/` path simultaneously serving English content and attempting client-side language detection for redirect, conflating two separate concerns in one page.

With the site growing to include a full landing page experience, it became clearer that `/` should be its own page—a language-selection LP—detached from any specific locale's content.

## Decision

### 1. All locale content moves under `/{locale}/`

Every piece of locale-specific content is placed under its locale prefix directory:

| Old path                          | New path                   |
| --------------------------------- | -------------------------- |
| `src/pages/blog/`                 | `src/pages/en/blog/`       |
| `src/pages/about.astro`           | `src/pages/en/about.astro` |
| `src/pages/index.astro` (EN home) | `src/pages/en/index.astro` |

Rendered URLs follow the same pattern:

| Old URL                  | New URL            |
| ------------------------ | ------------------ |
| `/blog/{slug}/`          | `/en/blog/{slug}/` |
| `/about`                 | `/en/about`        |
| `/` (serving EN content) | `/en/`             |

### 2. `getLocalizedPath()` always prefixes

The special-case for `defaultLang` is removed. The function now consistently returns `/{locale}/{path}` for all locales, including `en`.

```typescript
// Before
if (locale === defaultLang) return `/${cleanPath}`;
return `/${locale}/${cleanPath}`;

// After
return `/${locale}/${cleanPath}`;
```

### 3. Slug-stripping logic is uniform across all locale blog routes

All three `[...slug].astro` files now use the same pattern:

```typescript
params: { slug: post.id.replace(new RegExp(`${locale}$`), "") },
```

### 4. The root `/` becomes a standalone language-selection landing page

`src/pages/index.astro` is rewritten as a complete standalone HTML document (no `BaseLayout`, no auto-redirect). It provides:

- A full-viewport hero with an animated name display cycling through "Kang Daewook", "강대욱", and "姜待旭".
- A language-selection CTA with three cards (`/en/`, `/ja/`, `/ko/`).
- Its own `<head>` including an inline FOUC-prevention theme script and CSS keyframes, since `BaseHead` is not used.

### 5. RSS feed link format updated

`src/pages/rss.xml.js` now generates post links as `/${lang}/blog/${slug}/` to match new URL structure.

## Consequences

### Positive

- File structure under `src/pages/` is now symmetric: all locale content lives under `src/pages/{locale}/`, making it trivially navigable.
- `getStaticPaths()` slug logic is identical across all three locale blog routes; reduced maintenance surface and no more risk of divergence.
- `getLocalizedPath()` has no special cases; uniform behaviour reduces cognitive overhead.
- `/` is purpose-built as a marketing LP, cleanly separated from locale-specific content.
- Adding a fourth locale requires only creating `src/pages/{locale}/` with no asymmetric routing workarounds.

### Negative / Trade-offs

- All existing English deep-link URLs (e.g. `/blog/first-post/`) break and return 404. GitHub Pages does not support server-side redirects, so legacy links cannot be transparently redirected without a client-side redirect page or a custom 404.html strategy.
- `astro.config.mjs` retains `prefixDefaultLocale: false` in the `i18n` config block, which is now technically vestigial since pages are manually placed—not auto-routed—under `/en/`. This mismatch is low-risk but could cause confusion if Astro's i18n routing helpers are used in the future.

## Alternatives Considered

| Option | Reason not chosen |
| --- | --- |
| Keep English at root, add `/en/` aliases | Requires maintaining two sets of pages and doubles build output for EN content |
| Use Astro `i18n.routing.prefixDefaultLocale: true` | Would auto-generate the prefixed routes, but still requires custom LP at `/`; adds coupling to Astro i18n routing internals we are already managing manually |
| Auto-redirect from `/` to browser-detected locale | Implemented previously; removed because it gives no user choice, breaks back-button flow, and requires JavaScript to function |

## Evidence

- `src/pages/en/` directory (English pages now under locale prefix)
- `src/pages/en/blog/[...slug].astro` (symmetric slug-stripping pattern)
- `src/pages/index.astro` (standalone LP, no BaseLayout)
- `src/i18n/utils.ts` — `getLocalizedPath` (defaultLang special-case removed)
- `src/pages/rss.xml.js` (updated link generation)
- `src/components/HomeContent.astro` (blog links updated to `/${lang}/blog`)
